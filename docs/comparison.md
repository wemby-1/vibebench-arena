# Comparison: Where VibeBench Arena Fits

VibeBench Arena is a local-first quality console for AI-assisted coding evidence. It does not replace tests, does not prove semantic correctness by itself, and does not claim to beat benchmark suites.

Its job is narrower and practical: turn AI-generated changes into inspectable artifacts, summaries, risk signals, comparison outputs, and release/readiness evidence that humans can review before trusting a change.

## Comparison Matrix

| Category | Primary job | What it proves | What it misses | Where VibeBench fits |
| --- | --- | --- | --- | --- |
| Traditional CI | Run configured commands on a branch or pull request. | Whether the configured commands passed in that environment. | The surrounding review packet: risk context, artifact inventory, comparison summaries, and release readiness framing. | Packages local and CI evidence around an AI-assisted change so reviewers can inspect more than pass/fail. |
| Unit tests and linters | Catch known functional, style, and static-analysis issues. | Whether the project passes the checks the team wrote or configured. | Uncovered behavior, product intent, broad diff risk, review narrative, and release readiness context. | Runs alongside tests and linters, then records their results as part of a broader evidence trail. |
| GitHub Actions summaries | Present workflow output in GitHub. | What a workflow chose to print or attach. | Local-first inspection, project-level risk scoring, comparison outputs, and release audit records. | Produces GitHub-readable summaries while also keeping artifacts inspectable from the checkout. |
| AI code review assistants | Comment on code or suggest review findings. | Possible issues inferred by the assistant. | A reproducible local artifact trail and explicit release readiness checks. | Complements assistant feedback with concrete checks, risk signals, manifests, and review packets. |
| Chatbot / RAG applications | Answer questions or retrieve context through conversation. | The answer generated by the system for a prompt. | Repository-state evidence, configured check results, diff risk, and local release records. | Sits outside the chat transcript and records what happened in the project. |
| SWE-bench / AgentBench-style leaderboards | Evaluate agents or models against benchmark tasks. | How a model or agent performs on selected benchmark tasks. | Whether a developer's own local project change is ready to review, merge, or release. | Helps teams inspect their own AI-assisted changes rather than ranking agents. |
| Evaluation harnesses | Run structured experiments or task suites. | Results for the tasks, metrics, and environment the harness defines. | Day-to-day review artifacts, GitHub-ready summaries, and release readiness records for ordinary repo work. | Provides a practical review console for project changes, not a replacement for research evaluation. |
| Manual code review checklists | Guide human reviewers through expected questions. | That a reviewer considered checklist items when used carefully. | Automatically collected artifacts, machine-readable outputs, and repeatable local commands. | Turns checklist concerns into concrete evidence that reviewers can inspect and discuss. |

## Why not just CI?

CI is essential. VibeBench does not replace it.

CI usually answers whether configured commands passed. That is necessary, but an AI-assisted change often needs more review context: what changed, which risks appeared in the diff, which artifacts were generated, how the latest run compares with prior evidence, and whether local release readiness checks surfaced concerns.

VibeBench tries to package that review evidence around the change. It can run locally before a pull request, emit CI-readable outputs later, and leave Markdown/JSON artifacts that humans can inspect.

## Why not just a benchmark?

Benchmarks and leaderboards evaluate agents, models, or systems against task suites. They are useful for comparing capabilities under benchmark conditions.

VibeBench has a different job. It helps developers and maintainers review their own local project changes after an AI coding agent edits a real repository. The output is not a leaderboard rank; it is a review packet with checks, summaries, artifacts, risk signals, comparisons, and release/readiness evidence.

## What VibeBench Does Not Prove

VibeBench does not prove that generated code is semantically correct. It does not prove that a product decision is right, that security review is complete, or that a change should be merged automatically.

The evidence is only as strong as the configured checks, the risk rules, the repository context, and the human review that follows.
