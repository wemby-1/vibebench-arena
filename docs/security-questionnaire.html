Questionnaire scope

This questionnaire is project-maintained documentation, not a third-party audit, security certification, or compliance certification. Local users should review artifacts before publishing or sharing them.

Adopter Q&A

Does VibeBench require cloud services?

No. VibeBench is local-first for normal evaluation. GitHub Actions is optional CI infrastructure.

Does VibeBench upload source code?

The local CLI does not upload source code. GitHub Actions may upload configured artifacts from a CI run.

Does local evaluation require secrets?

No. Local evaluation does not require secrets or tokens.

What files does the Proof packet contain?

proof.html, proof.json, proof.md, proof-manifest.json, and optional proof.zip.

What files does the Static site preview contain?

index.html, showcase.html, site-check.json, site-preview.md, and optional site-preview.zip.

What files does the Evidence-room contain?

A landing page, review hub, reviewer guide, Trust Center, Security Questionnaire, scorecard, share-check.json, share-check.md, evidence summaries, nested Proof packet, and nested Static site preview. Open index.html first, then inspect share-check.md for the local pre-sharing scan summary.

What does GitHub Actions upload?

Downloadable proof packet, site preview, and evidence-room artifacts when CI is configured to run them.

Are generated HTML reports self-contained?

Yes. Generated HTML reports are intended to be self-contained static HTML.

Are remote resources allowed?

No. Generated static reports should avoid remote resources, external assets, scripts, and image dependencies.

Are absolute local paths allowed in public HTML?

No. Public/static HTML outputs should use relative links or placeholders.

Is JSON stdout pure JSON?

Yes. Commands with --json are expected to write machine-readable pure JSON stdout.

Is this SOC 2 or ISO 27001 certification?

No. VibeBench is not claiming SOC 2 certification or ISO 27001 certification.

Has there been an independent third-party audit?

No. VibeBench is not claiming an independent third-party audit.

How should sensitive repositories be handled?

Run locally first, avoid secrets, inspect generated artifacts, and review files before sharing.

Reviewer-friendly commands

python3 -m vibebench evidence-room --output-dir PATH --zip
python3 -m vibebench evidence-room --verify PATH
python3 -m vibebench evidence-room --verify PATH/evidence-room.zip
python3 -m vibebench proof --verify PATH/proof-packet
python3 -m vibebench site-preview --verify PATH/site-preview
python3 -m vibebench site-check
python3 -m vibebench ci --dry-run --json
python3 -m vibebench release-check
python3 -m vibebench doctor --strict

What the project does not claim

  • VibeBench is not claiming SOC 2 certification.
  • VibeBench is not claiming ISO 27001 certification.
  • VibeBench is not claiming an independent third-party audit.
  • VibeBench does not replace human review or provide a sandbox.
  • VibeBench does not automatically publish packages, create releases, enable GitHub Pages, or change repository settings.

Sharing and reporting

Review CI artifacts before sharing

Download artifacts, open the Evidence-room from index.html, inspect the Proof packet and Static site preview, verify the package, and run python3 -m vibebench share-check PATH.

Security issues

Use the repository Security Policy for vulnerability reporting expectations.

python3 -m vibebench share-check PATH
python3 -m vibebench share-check PATH --json

Evidence rooms include share-check.json and share-check.md. Share-check is a local pre-sharing aid, not a security certification, not a third-party audit, and not a guarantee. Users should still manually review artifacts before publishing.
